# Теги и блоки заказчика: как с админки вставляется информация на сайт

Краткий обзор механизмов, которые позволяют менять контент сайта из админки без правки кода. Код не трогаем — только структура и описание.

---

## 1. Config (Constance) — настройки сайта

**Что это:** глобальные настройки, которые редактируются в админке (раздел Constance / «Конфиг»). В шаблонах доступны как переменная `config`.

**Как подключается:** контекст-процессор `constance.context_processors.config` добавляет `config` в каждый шаблон.

**В шаблонах:**
- `{{ config.НАЗВАНИЕ_ПОЛЯ }}` — значение настройки (например `config.SITE_NAME`, `config.HEDER_LOGO_FILE`);
- `{% config_value 'НАЗВАНИЕ_ПОЛЯ' as переменная %}` — то же значение, но можно сохранить в переменную (тег из `utils_tags`).

**Примеры полей:** название сайта, логотип, иконка корзины, контакты в футере, копирайты, текст кнопок, включение/выключение блоков (телефон, соцсети и т.д.). Всё это заказчик меняет в админке, код не трогается.

---

## 2. Statictext — статические тексты по ключу

**Что это:** текстовые блоки (HTML разрешён), хранятся в БД. У каждого блока есть **ключ** (key). В шаблоне вызываешь по ключу — подставляется текст из админки.

**Где в админке:** приложение **Statictext** (статические тексты).

**Теги (statictext_tags):**
- `{% render_statictext 'ключ' %}` — выводит блок в месте вызова (с обёрткой/кэшем по настройкам);
- `{% render_statictext_var 'ключ' as переменная %}` — не выводит, а записывает текст в переменную (удобно для модалок, футера и т.д.).

**Примеры:** текст в футере, текст для cookie-модалки (`cookie-modal__text`), блоки на статичных страницах. Заказчик меняет текст в админке — на сайте обновляется автоматически.

---

## 3. Меню (Menu) — пункты меню из админки

**Что это:** меню с пунктами (название, ссылка, иконка, открытие в новой вкладке и т.д.). У каждого меню есть **ключ** (key). В шаблоне подставляешь меню по ключу.

**Где в админке:** приложение **Menu** (модели Menu, MenuItem).

**Теги (menu_tags):**
- `{% render_menu 'ключ_меню' %}` — выводит меню (список ссылок). Опции: `show_title=True`, `mmenu_mode=True` для мобильного меню.

**Важные ключи (создаются миграциями):**
- `header_menu` — пункты в шапке сайта (десктоп);
- `footer_customer_menu` — колонка «Покупателям» в футере.

В шаблоне хедера используется не тег, а переменная контекста: `header_menu` подставляется тегом `render_header` из того же меню по ключу `header_menu`. То есть логика «из админки» та же — просто вывод через цикл `{% for menu_item in header_menu %}`.

## 4. Контейнеры тегов (SliderTagContainer) — блоки из «кирпичиков»

**Что это:** именованные зоны на странице (например «над шапкой», «под шапкой», «низ страницы»). В каждой зоне заказчик в админке собирает список **блоков**. Каждый блок — это вызов одного из разрешённых тегов (слайдер, статичный текст, каталог, баннер и т.д.) с опциональными аргументами и оформлением (фон, отступы, заголовок).

**Где в админке:** **Customization** → **Контейнеры тегов** (SliderTagContainer). У контейнера — ключ и название. Внутри — элементы (SliderTagContainerItem) с полями: какой тег вызвать (`tag_line`), аргументы (`args`), обёртка, фон, только главная и т.д.

**Тег (customization_tags):**
- `{% render_tag_container 'ключ_контейнера' wrapper_main_class="доп. класс" %}` — выводит все включённые блоки этого контейнера по порядку.

**Ключи контейнеров (из кода):**
- `pre_header` — блоки перед шапкой (сквозные);
- `post_header` — блоки после шапки (сквозные);
- `middle` — блоки главной страницы;
- `bottom` — блоки нижней части страницы;
- `staticpage_container` — блоки для статичных страниц (контейнер).

В `base.html` используются, например: `render_tag_container 'pre_header'`, `render_tag_container 'post_header'`, `render_tag_container 'bottom'`.

**Какие теги можно вставить в контейнер:** список строится автоматически из всех приложений проекта — все теги, в имени которых есть `render_` (например `catalog_tags:render_catalog_categories_menu_simple`, `statictext_tags:render_statictext`, `slide_tags:render_big_slides`). Конкретный список смотри в админке в выпадающем списке «tag function name» у элемента контейнера.

Итог: заказчик в админке выбирает, какие блоки (какой тег + параметры) показывать в каждой зоне и в каком порядке — без правки шаблонов.

## 5. Свои стили и скрипты (CustomStyle / CustomScript)

**Что это:** загрузка дополнительных CSS/JS из админки (файлы или инлайн-код). Нужны, чтобы заказчик мог подключать свои стили (например общий `style.css`) или скрипты без деплоя.

**Где в админке:** **Customization** → свои стили / свои скрипты.

**Теги (customization_tags):**
- `{% render_custom_styles %}` — выводит ссылки на файлы и/или инлайн-стили (подключается в base.html в блоке css);
- `{% render_custom_scripts %}` — то же для скриптов.

Логика «подключить что задано в админке» — в шаблоне только вызов тега, содержимое полностью из админки.

---

## 6. Кастомный шаблонный блок (CustomTemplateBlock)

**Что это:** блок с произвольным кодом Django-шаблона (HTML + теги), хранится в БД по ключу. Рендерится с общим контекстом страницы, поэтому доступны `request`, `config` и т.д.

**Где в админке:** **Customization** → кастомные шаблонные блоки (CustomTemplateBlock). Поля: key, name, code (текст шаблона).

**Тег (customization_tags):**
- `{% render_custom_template_block 'ключ' %}` — рендерит сохранённый шаблонный код и выводит результат (через шаблон `custom-template-block.html`).

Используется реже; даёт заказчику возможность вставить свой фрагмент разметки/логики без деплоя (осторожно с безопасностью и ошибками в коде).

---

## 7. Кастомная навигация (CustomNavigation)

**Что это:** отдельные блоки навигации с набором ссылок (название, URL, картинка/SVG). Отличается от Menu: привязан к блоку с оформлением (фон, отступы), ссылки с картинками.

**Где в админке:** **Menu** → Custom Navigation / Custom Link.

**Тег (menu_tags):**
- `{% render_custom_navigation 'ключ' %}` — выводит блок навигации по ключу (шаблон `custom-navigation.html`).

---

## 8. Кастомные формы (Custom forms)

**Что это:** формы, собранные в админке (поля, типы, порядок). Выводятся на сайт по ключу формы.

**Тег (custom_form_tags):**
- В шаблоне обычно используется вставка формы внутри статичного текста: в Statictext включается «use custom form» и в текст вставляется `{% custom_form "КЛЮЧ-ФОРМЫ" %}`. Либо форма подключается отдельно там, где нужна (например форма обратной связи, «Перезвоните мне»).

Логика «что показывать» задаётся в админке (структура формы и ключ), в коде — только вызов тега/включения.

---

## Сводка: где что лежит

| Что нужно вставить с админки | Механизм | Тег / переменная |
|------------------------------|----------|-------------------|
| Название сайта, логотип, контакты, кнопки | Config (Constance) | `config.ПОЛЕ` или `{% config_value 'ПОЛЕ' %}` |
| Текст по ключу (футер, cookie, страницы) | Statictext | `{% render_statictext 'key' %}`, `{% render_statictext_var 'key' as var %}` |
| Меню в шапке / футере | Menu | `{% render_menu 'footer_customer_menu' %}`, в хедере — `header_menu` из `render_header` |
| Зоны с блоками (слайдеры, текст, каталог и т.д.) | SliderTagContainer | `{% render_tag_container 'pre_header' %}`, `'post_header'`, `'bottom'`, `'middle'` |
| Свои CSS/JS | CustomStyle / CustomScript | `{% render_custom_styles %}`, `{% render_custom_scripts %}` |
| Произвольный кусок шаблона | CustomTemplateBlock | `{% render_custom_template_block 'key' %}` |
| Блок навигации с картинками | CustomNavigation | `{% render_custom_navigation 'key' %}` |
| Формы (обратная связь, заявки) | Custom forms + Statictext или отдельно | `{% custom_form "KEY" %}` внутри текста или отдельный вывод формы |

Чтобы «подключить теги заказчика» в новом дизайне (например в хедере/футере 8 марта): используй те же теги и переменные (`config`, `render_statictext_var`, `render_menu`, `location_contact_phone` и т.д.) в новых шаблонах — логика остаётся из админки, меняется только вёрстка и при необходимости набор/порядок вызовов.
